### 4. 数据映射器 ✅

**文件：** `src/lib/sync/org-sync-mapper.ts`

**核心功能：**
- 将EKP组织元素映射为本系统格式
- 将EKP人员映射为本系统格式
- 数据清洗（过滤无效数据、离职人员）
- 统计各类型数据数量
- 比较数据是否需要更新

**映射规则：**
- EKP类型 → 本系统类型：org=1, dept=2, post=3, group=4
- EKP性别 → 本系统性别：M=1, F=2
- 人员默认密码：123456（可配置）
- 一人多岗：支持 posts 数组映射

### 5. 同步服务核心逻辑 ✅

**文件：** `src/lib/sync/org-sync.service.ts`

**核心功能：**
- ✅ 全量同步（手动触发）
- ✅ 增量同步（定时+手动）
- ✅ 分页处理（防止内存溢出）
- ✅ 对比更新（以EKP为准）
- ✅ 软删除处理（全量同步时）
- ✅ 批量处理（每批500条）
- ✅ 错误处理和重试

**同步流程：**
```
1. 检查是否有正在运行的同步
2. 创建同步日志记录
3. 调用EKP接口获取数据
4. 数据清洗和过滤
5. 分批处理组织元素
6. 分批处理人员
7. 全量同步时处理软删除
8. 更新同步日志
9. 保存同步明细
10. 更新时间戳令牌
```

### 6. API接口 ✅

**文件：**
- `src/app/api/org-sync/route.ts`
- `src/app/api/org-sync/logs/[...id]/route.ts`

**提供的接口：**

#### 1. 触发同步
```bash
POST /api/org-sync
Content-Type: application/json

{
  "syncType": "full",           // 或 "incremental"
  "operatorId": "user-id",
  "operatorName": "用户名",
  "returnOrgType": [{"type":"org"},{"type":"dept"},{"type":"post"},{"type":"person"}]
}
```

#### 2. 获取同步状态
```bash
GET /api/org-sync?action=status
```

#### 3. 查询同步日志列表
```bash
GET /api/org-sync/logs?syncType=full&status=completed&page=1&pageSize=20
```

#### 4. 获取同步日志详情
```bash
GET /api/org-sync/logs/{syncLogId}
```

#### 5. 获取同步明细
```bash
GET /api/org-sync/logs/{syncLogId}?details=true
```

## ⚠️ 注意事项

### 1. 同步并发控制

- 同时只能有一个同步任务运行
- 如果有任务正在运行，新任务会被拒绝
- 前端需要根据返回状态提示用户

### 2. 数据一致性

- 全量同步时使用软删除（fd_is_available=0）
- 增量同步时根据 isAvailable 字段判断是否软删除
- 以EKP数据为准，本地修改会被覆盖

### 3. 性能考虑

- 大数据量时采用分批处理（默认500条/批）
- 使用批量插入减少数据库操作
- 建议在业务低峰期执行全量同步

### 4. 错误处理

- 单条数据失败不影响整体同步
- 所有错误都会记录到同步明细表
- 前端需要展示详细的错误信息

### 5. 密码处理

- 同步过来的人员使用默认密码 123456
- 首次登录后建议用户修改密码
- 默认密码可在数据库配置表中修改
