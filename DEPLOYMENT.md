# 部署配置说明

## 问题说明

### 为什么每次部署后配置会丢失？

当前架构存在以下问题：

1. **内存缓存丢失**：数据库连接配置缓存在内存中（`lastActiveConfig`），部署后进程重启导致缓存丢失
2. **无启动时自动重连**：应用启动时不会自动从数据库读取配置并重新连接
3. **自动重连依赖用户访问**：只有访问 `/api/database` 时才会触发自动重连

### 数据存储架构

```
┌─────────────────────────────────────────┐
│  应用层（每次部署重启）                    │
│  - 内存缓存: lastActiveConfig ❌         │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│  MySQL 数据库（持久化）✅                 │
│  - database_configs 表                   │
│  - api_keys 表                           │
│  - ekp_configs 表                        │
│  - 其他业务数据                          │
└─────────────────────────────────────────┘
```

## 解决方案

### 方案1：使用环境变量配置数据库（推荐）

#### 优点
- ✅ 部署后自动连接，无需手动操作
- ✅ 配置不依赖数据库状态
- ✅ 适合生产环境

#### 配置步骤

1. **在部署平台配置环境变量**

   如果使用 Vibe Coding 平台，在部署配置中添加以下环境变量：

   ```
   DB_HOST=your_mysql_host
   DB_PORT=3306
   DB_USER=your_mysql_user
   DB_PASSWORD=your_mysql_password
   DB_NAME=future_office
   ```

2. **配置花生壳映射（如果使用远程数据库）**

   ```
   花生壳域名 → MySQL 服务器
   例如: abc123.oray.com → 192.168.1.100:3306
   ```

3. **部署后自动连接**

   - 应用启动时会自动读取环境变量
   - 自动测试连接并连接到数据库
   - 自动同步配置到 `database_configs` 表

#### 注意事项

- 环境变量优先级最高
- 如果环境变量配置成功，会自动同步到数据库
- 如果环境变量连接失败，会尝试从数据库读取之前的配置

### 方案2：启动时自动重连（已实现）

#### 优点
- ✅ 不需要额外配置
- ✅ 自动从数据库读取配置
- ✅ 适合已有数据库配置的场景

#### 工作原理

```
应用启动
  ↓
initializeApp() 执行
  ↓
检查环境变量配置 → 有 → 自动连接
  ↓
无 → 读取 database_configs 表
  ↓
找到激活配置 → 测试连接 → 连接成功
  ↓
未找到配置 → 提示手动初始化
```

#### 使用方法

1. 首次部署后，访问管理员后台
2. 进入 `/admin/database` 页面
3. 点击"初始化数据库"
4. 填写数据库连接信息
5. 初始化成功后，配置会保存在 MySQL 中
6. 下次部署时，会自动重新连接

### 方案3：手动重新连接

#### 适用场景
- 开发环境
- 测试环境
- 数据库配置经常变化

#### 操作步骤

1. 部署后访问 `/admin/database`
2. 查看数据库连接状态
3. 如果未连接，点击"连接"按钮
4. 选择之前的配置并连接

## 部署流程建议

### 首次部署（全量部署）

1. **准备工作**
   - 确保远程 MySQL 数据库可访问
   - 配置花生壳映射（如果使用远程数据库）

2. **选择配置方式**
   - 推荐：使用环境变量配置
   - 备选：手动初始化

3. **部署应用**
   - 上传代码
   - 配置环境变量（如使用方案1）
   - 启动应用

4. **验证连接**
   - 访问 `/admin/database`
   - 检查数据库连接状态
   - 测试查询功能

### 后续部署（增量部署）

**理论上可以实现增量部署，但实际操作中：**

- ❌ 增量部署无法解决数据库连接问题（配置不是代码）
- ❌ 增量部署需要复杂的版本管理和依赖分析
- ✅ 全量部署 + 自动重连 更简单可靠

**推荐部署流程：**

1. 使用 Git 管理代码版本
2. 部署时使用全量包（完整代码）
3. 依赖数据库持久化存储配置
4. 应用启动时自动重连

### 配置优先级

```
1. 环境变量配置（最高优先级）
   ↓
2. 数据库配置表
   ↓
3. 未配置（需要手动初始化）
```

## 常见问题

### Q1: 部署后数据库连接失败？

**可能原因：**
- 花生壳映射失效
- MySQL 服务未启动
- 网络连接问题
- 密码错误

**解决方法：**
1. 检查花生壳映射是否正常
2. 测试 MySQL 是否可访问：
   ```bash
   mysql -h your_oray_domain -u user -p
   ```
3. 查看应用日志：`/app/work/logs/bypass/app.log`

### Q2: 为什么配置丢失了？

**原因：**
- 内存缓存丢失（正常现象）
- 数据库中的配置仍在（不会丢失）

**解决方法：**
1. 应用启动时会自动重连
2. 如未自动重连，访问 `/admin/database` 手动连接

### Q3: 数据库中的业务数据会丢失吗？

**不会丢失！**

- 业务数据存储在 MySQL 中，与部署无关
- 部署只影响应用代码，不影响数据库
- 除非手动执行 `DROP DATABASE` 操作

### Q4: 可以实现增量部署吗？

**技术上可行，但不推荐：**

**增量部署的问题：**
- 需要复杂的依赖分析
- 可能导致版本冲突
- 配置管理复杂
- 难以回滚

**推荐方案：**
- 全量部署（简单可靠）
- 数据库持久化（配置保留）
- 自动重连机制（无需手动操作）

### Q5: 如何确保配置不丢失？

**最佳实践：**

1. **使用环境变量**（推荐）
   - 配置在部署平台
   - 每次部署自动应用

2. **备份数据库配置**
   ```sql
   -- 导出 database_configs 表
   mysqldump -u user -p future_office database_configs > db_config_backup.sql
   ```

3. **定期备份数据库**
   ```sql
   -- 导出整个数据库
   mysqldump -u user -p future_office > full_backup.sql
   ```

## 总结

### 核心要点

1. **配置不会真正丢失**：存储在 MySQL 中，部署不受影响
2. **自动重连已实现**：启动时自动从数据库读取配置
3. **环境变量优先**：推荐在生产环境使用
4. **增量部署不推荐**：全量部署 + 数据持久化更可靠

### 推荐部署方案

```
生产环境:
├── 使用环境变量配置数据库
├── 部署时应用自动连接
└── 无需手动操作

开发环境:
├── 首次手动初始化数据库
├── 部署后自动重连
└── 配置持久化在 MySQL
```

### 技术架构优势

```
✅ 代码与数据分离
✅ 配置持久化
✅ 自动重连机制
✅ 环境变量支持
✅ 易于维护
```
